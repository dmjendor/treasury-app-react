## Tailwind and theming

- Styling uses Tailwind with theme color scales backed by CSS variables (OKLCH).
- Theme palettes include (but are not limited to):
  - `primary`
  - `accent`
  - `danger`
  - `warning`
  - `surface`
- Numeric steps (50â€“900) are used to create depth and contrast.

---

## Class usage rules

- DO NOT invent utility classes.
- If unsure whether a class exists, check: `/app/\_styles/globals.css`
- DO NOT use semantic aliases such as:
  - `text-fg`
  - `text-muted-fg`
  - `bg-card`
  - similar abstract tokens

---

## Color usage guidelines

- Always use explicit theme colors:
  - `text-primary-900` for dark text on light backgrounds
  - `text-primary-50` for light text on dark backgrounds
- Use appropriate numeric steps to create hierarchy:
  - headers: stronger contrast
  - secondary text: softer contrast
- Do not default everything to `primary`.
  - Use `accent`, `danger`, `warning`, etc. based on intent.

---

## Icons

- All icons must go through `IconComponent`.
- SVG icons should use `currentColor` for fill or stroke.
- Color is applied via text color utilities on the parent or icon element.
- Do not inline hard-coded colors inside SVGs.

---

## Buttons and links

- Buttons must use `Button` or `LinkButton`.
- Do not recreate button styles manually.
- Button variants control intent; colors may be overridden only when appropriate.

---

## Visual style goals

- Clean, minimal, dark-fantasy-inspired aesthetic.
- Subtle depth using:
  - borders
  - layered surfaces
  - restrained shadows
- Avoid loud glows or heavy gradients unless explicitly requested.

---

## Layout and interaction

- Prefer clarity over cleverness.
- Avoid layout shifts when toggling UI state.
- Use columns or panels instead of expanding grid cards when editing.
- Animations, if used, must be subtle and purposeful.

---

## When in doubt

- Ask before guessing:
  - available color scales
  - valid numeric steps
  - background brightness (light vs dark surface)
- Do not assume a class or token exists.
